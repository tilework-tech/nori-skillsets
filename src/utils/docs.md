# Noridoc: utils

Path: @/src/utils

### Overview

Shared utility functions used across the codebase for URL manipulation, filesystem path resolution, network proxy support, API token parsing, non-interactive environment detection, error classification, upload filename filtering, symlink-aware filesystem type checking, and safe (corruption-refusing, atomic) mutation of shared JSON state files. These are pure helpers with no business logic or state of their own.

### How it fits into the larger codebase

Every layer of the application imports from this module. The API layer uses `url.ts` for URL construction, `fetch.ts` for error types and proxy initialization, and `apiToken.ts` for parsing the orgId embedded in API tokens during per-request auth resolution. The CLI config system uses `path.ts` for resolving installation directories, `home.ts` for finding the user's home directory, and `apiToken.ts` when computing whether a stored API token matches a target registry org. The login command uses `apiToken.ts` to validate `--token` input. The CLI entrypoint at `@/src/cli/nori-skillsets.ts` uses `nonInteractive.ts` to set the default value of the global `--non-interactive` flag, which propagates to every subcommand via `program.opts()`. The `fetch.ts` error classes (`NetworkError`, `ApiError`, `SkillCollisionError`) are the canonical error types used throughout the codebase. The `dirent.ts` helpers are used by skillset discovery (`@/src/norijson/skillset.ts`), the upload flow's inline detection (`@/src/cli/commands/registry-upload/registryUpload.ts`), and the shared tarball builder (`@/src/packaging/archive.ts`) wherever `fs.readdir({ withFileTypes: true })` results need to follow symlinks.

### Core Implementation

**`url.ts`** handles URL normalization (preventing double slashes), org ID validation/extraction, and URL construction for three Nori domains: `*.tilework.tech` (Watchtower), `*.nori-registry.ai` (private registries), and `*.noriskillsets.dev` (public/org registries, where `"public"` maps to the apex domain). It also provides the reference-resolution pair: `parseNamespacedPackage()` parses a `"org/package@version"` spec into `{ orgId, packageName, version }` (a bare, non-namespaced spec is treated as the `"public"` org), and `namespacedName()` maps that parsed reference to its on-disk skillset name: it returns the bare `packageName` for the public registry (stored flat under `profiles/<name>`) and `"<orgId>/<packageName>"` for an org (stored nested under `profiles/<orgId>/<name>`). Callers that must record or look up a skillset by its local name (e.g. `@/src/cli/commands/registry-install/registryInstall.ts`) parse with the former and derive the name with the latter rather than string-splitting themselves.

**`fetch.ts`** initializes HTTP proxy support via undici's `EnvHttpProxyAgent` (call `initializeProxySupport()` once at startup), and defines four custom error classes: `NetworkError` (connectivity issues with proxy-aware messages), `ApiError` (HTTP status errors), `SkillCollisionError` (409 conflicts during skillset upload with per-skill conflict details), and `SubagentCollisionError` (409 conflicts during skillset upload with per-subagent conflict details, structurally mirroring `SkillCollisionError`). The `formatNetworkError()` function produces user-friendly messages that vary based on error code and proxy configuration.

**`path.ts`** resolves installation directories through a priority chain: CLI flag > config value > home directory, with provenance tracking via the `ResolvedInstallDir` type (`{ path: string; source: InstallDirSource }`). The `InstallDirSource` type (`"cli" | "config" | "default"`) records where the install directory was resolved from, allowing downstream code to derive behaviors like skipping manifest operations or config persistence when the source is a transient CLI override (`"cli"`). The `resolveInstallDir()` function accepts `cliInstallDir` and `configInstallDir` as separate string parameters (rather than a full config object) and returns a `ResolvedInstallDir`. The `normalizeInstallDir()` helper handles tilde expansion, relative-to-absolute conversion, and strips trailing agent config directory segments (e.g., `.claude`) when `agentDirNames` is provided.

**`home.ts`** returns the home directory with override support: `NORI_GLOBAL_CONFIG` env var (for test isolation) > `os.homedir()` > `process.env.HOME`.

**`apiToken.ts`** parses and validates API tokens. API tokens follow the format `nori_<orgId>_<64 hex chars>`, where the orgId is embedded in the token itself so no separate orgId flag or env var is needed. Exports `API_TOKEN_PATTERN` (the canonical regex), `isValidApiToken({ token })` (shape check), and `extractOrgIdFromApiToken({ token })` (returns the orgId or `null` for malformed input). Call sites in `@/src/cli/config.ts`, `@/src/api/base.ts`, `@/src/api/registryAuth.ts`, and `@/src/cli/commands/login/login.ts` use these helpers whenever an API token needs to be matched against a target registry URL.

**`uploadFileFilter.ts`** is the single source of truth for paths that the upload tarball builder must skip. It exports two functions:

- `shouldExcludeFromUpload({ relativePath, cargoManifestDirs? })` takes the file's path relative to the package root (both `/` and `\` separators accepted) rather than a bare basename. It excludes individual junk files by basename — the local download-metadata file (`.nori-version`), Vim swap files, editor backup files (trailing `~`), macOS `.DS_Store` and AppleDouble resource forks (`._*`), and Windows `Thumbs.db` — and additionally excludes dependency/build "bloat" directories by inspecting the path's directory segments.
- `collectCargoManifestDirs({ relativePaths })` precomputes the set of directories that contain a `Cargo.toml` sibling, returned as forward-slash-joined relative paths (the empty string represents the package root). The tarball builder calls this once over its full file listing and passes the result into `shouldExcludeFromUpload` so a `target/` directory can be recognized as Rust build output.

Bloat-dir exclusion only matches **non-terminal** (directory) segments of the path, never the leaf file: `node_modules/` and `.venv/` are excluded at any nesting depth, while `target/` is excluded only when its parent directory appears in `cargoManifestDirs` (precise Rust build-output detection, never a blind guess). Exact-segment matching means lookalikes (`node_modules_old`, `my.venv`, `target_old`) and leaf files that merely share one of these names are kept. When `cargoManifestDirs` is omitted, no `target/` directory is excluded. The sole consumer is `createArchive` in `@/src/packaging/archive.ts`, the shared tarball builder behind both upload commands (`@/src/cli/commands/registry-upload/registryUpload.ts` and `@/src/cli/commands/skill-upload/skillUpload.ts`), so editor/OS junk and dependency/build output never enter the registry's content hash and never trigger spurious "content differs" conflicts on re-upload.

**`dirent.ts`** exports `isDirentDirectory({ parentDir, entry })` and `isDirentFile({ parentDir, entry })`, which are symlink-aware replacements for `Dirent.isDirectory()` and `Dirent.isFile()`. Node.js `Dirent.isDirectory()` returns `false` for symlinks that point to directories, which breaks discovery of symlinked skillsets. These helpers first check the native Dirent method, and if the entry is a symbolic link, follow it with `fs.stat()` to determine the target type. They are the canonical way to check directory/file type from `Dirent` objects throughout the codebase; any code using `fs.readdir({ withFileTypes: true })` that must follow symlinks should use these instead of raw `entry.isDirectory()` / `entry.isFile()`.

**`jsonFile.ts`** is the shared discipline for editing a JSON file the CLI does not exclusively own (Claude Code's `settings.json` / `.claude.json`, the Nori config). It exports two primitives:

- `readJsonObjectFile({ filePath, ifAbsent })` returns the parsed object, but distinguishes "absent" from "corrupt": it returns `ifAbsent` **only** on `ENOENT` (and for an empty / whitespace-only file, which has no content to preserve — the classic crash-mid-write artifact), and **throws** ("Refusing to modify `<path>`: ... Fix or remove the file, then re-run.") when the file exists but is not valid JSON or does not parse to a plain object. This is the load-bearing invariant — callers write the result back, so treating a present-but-unparseable file as empty would erase the user's file. Every caller reads, mutates, and writes back, so the throw makes the whole read-modify-write abort instead of clobbering.
- `writeJsonFileAtomic({ filePath, value })` creates parent dirs, resolves an existing symlink destination, serializes to a unique sibling temp file beside the real write target (`<path>.tmp-<pid>-<random>`), then `rename()`s it into place. `rename(2)` is atomic within a filesystem, so a crash or a concurrent reader sees either the old file or the new one, never a truncated mix; the pid + random suffix keeps concurrent writers from colliding on a shared temp name, and symlinked config paths keep their symlink instead of being replaced by a regular file. When the target already exists, its permission mode is used when the temp file is created, before JSON bytes are written, so a tightened mode (e.g. `0600` on a secret-bearing `~/.nori-config.json` / `~/.claude.json`) is not widened during the write window. The temp file is chmodded back to the existing mode before rename for exact-mode preservation and cleaned up on failure.

Consumers are the JSON-state mutation sites the CLI shares with Claude Code: the claude-code hooks/statusline/announcements loaders (all edit `~/.claude/settings.json`) and the shared MCP loader's `mergeMcpServersIntoJson` (grafts into `~/.claude.json`) in @/src/cli/features/, plus `writeConfigFile`/`updateConfig` for `~/.nori-config.json` in @/src/cli/config.ts. These replaced hand-rolled `try { JSON.parse } catch { = {} }` + `fs.writeFile` patterns that silently overwrote corrupt files.

**`nonInteractive.ts`** exports the pure helper `isNonInteractiveEnvironment({ stdin?, stdout?, env? })`, which returns `true` when `env.CI` is set to a truthy value (anything other than the lowercase-insensitive strings `""`, `"0"`, or `"false"`), OR `stdin.isTTY` is not strictly `true` (piped stdin, cron, no controlling terminal), OR `stdout.isTTY` is not strictly `true` (our output is piped/captured). Interactivity therefore requires that BOTH `stdin` AND `stdout` be TTYs. All three inputs default to `process.stdin`, `process.stdout`, and `process.env` so the helper is trivially callable at process boot. The `stdout` check exists for the case where a parent process captures our stdout while inheriting the terminal's TTY stdin — notably nori-cli spawning `nori-skillsets list` and parsing its stdout line-by-line — where prompting would corrupt the captured output and block forever waiting for a keypress. Consumed by `@/src/cli/nori-skillsets.ts` as the default for the `-n, --non-interactive` global option and for the pre-parse argv peek that gates `checkForUpdateAndPrompt`. There is intentionally no escape hatch to force interactive mode under CI.

### Things to Know

The `NORI_GLOBAL_CONFIG` environment variable overrides the home directory globally, which affects config file location, skillset storage, and all path resolution. This is the primary mechanism for test isolation.

`SkillCollisionError` and `SubagentCollisionError` each carry structured conflict data (`conflicts` array with per-item resolution options) that the CLI upload flow uses to prompt users for resolution decisions. Both error classes have corresponding type guards (`isSkillCollisionError`, `isSubagentCollisionError`) that use both `instanceof` checks and discriminant property checks (`isSkillCollisionError` / `isSubagentCollisionError` booleans) for cross-boundary safety.

Each entry in those `conflicts` arrays (`SkillConflictInfo` / `SubagentConflictInfo`) carries an optional `fileChanges?: ReadonlyArray<FileChange> | null` field that the registrar populates on 409 upload responses. `FileChange` is `{ path, status: "added" | "modified" | "removed", isBinary, existingContent?, existingTruncated? }`, enumerating the paths that differ between the uploaded bundle and the existing latest version (metadata files like `nori.json` / `.nori-version` are excluded by the server to match the content-hash contract). `existingContent` is only populated for non-binary modified/removed entries within server-side size caps; when content is skipped or truncated, `existingTruncated` is set. Field is optional for backwards compatibility — pre-#331 registrars omit it and the CLI treats absence as "no per-file detail available". The type is mirrored in `@/src/api/registrar.ts` (`SkillConflict` / `SubagentConflict`) and is consumed by the upload flow via the `fileChangesFormat.ts` helpers in `@/src/cli/prompts/flows/`.

`jsonFile.ts`'s "refuse on corrupt" rule is a behavioral contract: a corrupt `~/.claude/settings.json`, `~/.claude.json`, or `~/.nori-config.json` now makes affected commands fail loudly rather than silently overwriting the file (which previously dropped user hooks/MCP entries or stored credentials). "Absent" is still the benign case that seeds a fresh object. Any new site that reads a shared JSON file, mutates it, and writes it back should use these primitives rather than re-rolling parse-with-empty-fallback.

The API token format is deliberately self-describing: the orgId is embedded between `nori_` and the hex key, separated by underscores. Because orgIds themselves may contain hyphens, the regex uses `_` as the delimiter rather than `-`. The orgId is parsed on demand at read/request time rather than being persisted separately — this means there is no stale-scope risk when a token is replaced, and callers never need to pass the orgId alongside the token.
