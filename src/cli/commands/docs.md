# Noridoc: commands

Path: @/src/cli/commands

### Overview

The commands directory contains all CLI command implementations for the `nori-skillsets` CLI tool. Each command lives in its own subdirectory with a main implementation file and tests. The top-level files handle command name mapping and Commander.js registration.

### How it fits into the larger codebase

The CLI entry point delegates to `noriSkillsetsCommands.ts`, which registers every command with a Commander.js `program` instance. Each registration function imports the `*Main` entry point from the corresponding subdirectory and wires it to Commander options/arguments. Commands depend on `@/cli/config.js` for configuration, `@/cli/features/` for agent integration and skillset management (including the install orchestration at @/src/cli/features/install/, imported statically), `@/api/` for registry API calls, `@/packaging/` for package mechanics (tarballs, atomic installs, `.nori-version` provenance, registry lookup — see @/src/packaging/docs.md), and `@/cli/prompts/flows/` for interactive user prompts.

### Core Implementation

`cliCommandNames.ts` defines the `CommandNames` type and a lookup table that maps logical command names (e.g., `download`, `externalSkill`) to their CLI string equivalents. This is consumed by commands that need to display command hints in error messages.

`noriSkillsetsCommands.ts` is the central registration hub and the **sole caller** of `@clack/prompts` `intro()` and `outro()`. It exports one `registerNoriSkillsets*Command` function per command, each taking a `{ program: Command }` argument. Many commands have hidden aliases (e.g., `switch` also registers `switch-skillset`, `switch-skillsets`, and `use`) to support both short and long forms.

**Lazy handler loading:** `noriSkillsetsCommands.ts` registers commander names, descriptions, options, and aliases eagerly at startup, but each command's `*Main` handler module is loaded lazily via `await import("@/cli/commands/<dir>/<handler>.js")` **inside** the `.action()` callback (or a shared action closure for commands with multiple aliases). This keeps `--help` and `--version` unchanged while keeping heavy handler dependencies (e.g., firebase/auth pulled in by `login.ts`, registry API clients, chokidar) out of the startup import graph for `nori-skillsets --version` and `nori-skillsets list-active`. Only `intro`/`outro` from `@clack/prompts`, `AgentRegistry`, and type-only symbols remain eager at the top of the file. New commands added to this file should follow the same pattern: register eagerly, `await import` the handler inside `.action()`. The `@/` specifier is rewritten identically by `tsc-alias` for both static and dynamic imports, so the build guard in `scripts/build.sh` (which fails if any `@/` imports survive) still passes.

The file defines `wrapWithFraming()`, which enforces the three-layer separation of concerns for CLI visual framing:

```
Registration layer (noriSkillsetsCommands.ts)      <-- intro() / outro() live here
    |
    +-- wrapWithFraming({ title, action, exitOnFailure?, silent? })
            |
            +-- calls intro(title) before the action (suppressed when silent)
            +-- calls action(), which returns { success, cancelled, message }
            +-- calls outro(result.message) after (suppressed when cancelled or silent)
            +-- calls process.exit(1) if exitOnFailure && !success
            +-- catches thrown errors, displays them via outro (suppressed when silent), and exits
```

When `silent` is true, `wrapWithFraming` suppresses all `intro()`/`outro()` output, including error messages. The `silent` flag is forwarded from the global `--silent` CLI option at the registration layer.

Commands that do not need visual framing (single-step, scriptable, or producing raw output) are called directly without `wrapWithFraming`. These include `clear`, `clear-current`, `completion`, `list`, `list-active`, `current`, `logout`, `dir`, `install-location`, and `watch stop`.

**install-location** (@/src/cli/commands/install-location/): Displays the resolved Nori installation directory from config. Supports `--non-interactive` (plain output for scripts). Uses `resolveInstallDir()` from @/src/utils/path.ts with the standard priority chain (CLI flag > config > home directory).

**list-active** (@/src/cli/commands/list-active/listActive.ts): Discovers which skillsets are currently active in the directory hierarchy. `findActiveSkillsets` resolves a start directory (from the global `--install-dir` option or `process.cwd()`), then walks upward to the filesystem root. At each directory level, it iterates all agents from `AgentRegistry.getInstance().getAll()` and reads the skillset name via `getInstalledSkillsetName` from @/src/cli/features/agentOperations.ts — the single `.nori-managed` marker read path. Results are deduplicated via `Set` and returned sorted. `listActiveMain` writes each skillset name to stdout one per line (raw `process.stdout.write`, no framing), or exits with code 1 if none are found. Hidden alias: `la`. This command only checks `.nori-managed` marker files and does not support legacy NORI-AI MANAGED BLOCK detection.

**switch-skillset** (@/src/cli/commands/switch-skillset/switchSkillset.ts): The `switchSkillsetAction` function handles skillset switching with local change detection and multi-agent broadcasting. The `name` argument is optional; when omitted, the command resolves it before proceeding:

1. **Resolve skillset name** (when `name` is not provided): In interactive mode, calls `listSkillsets()` from @/src/norijson/skillset.ts to discover installed skillsets, then presents a `select()` prompt (from `@clack/prompts`) for the user to choose. Throws if no skillsets are installed. In non-interactive mode, throws an error listing available skillsets with usage instructions.
2. **Detect local changes**: Calls `detectLocalChanges({ agent, installDir })` (from @/src/cli/features/agentOperations.ts) on the first default agent, which reads the (agent, install dir) manifest (with legacy fallbacks) and compares current agent directory file hashes against stored hashes. Because manifests are keyed per install directory, this works for `--install-dir` overrides too; directories Nori never installed to return null.
3. **Handle changes** (if detected):
   - With `--force` flag: skips change handling entirely and proceeds (discards local changes)
   - In non-interactive mode (without `--force`): throws an error (safe default prevents data loss)
   - In interactive mode: displays modified/added/deleted files and prompts user to choose:
     - Proceed anyway (changes will be lost)
     - Save current config as new skillset first (broadcasts `captureExistingConfig()` to ALL default agents)
     - View changes (see content-level diffs of modified files; only shown when modified files exist)
     - Abort
4. **Confirm switch**: Prompts user to confirm the skillset switch
5. **Execute switch**: In non-interactive mode, broadcasts the switch to all configured `defaultAgents` (loops over all agents). In interactive mode, delegates to `switchSkillsetFlow` which also broadcasts to all resolved agents. Both paths explicitly pass `skillset: name` to `installMain`. The `skillset` argument is required because `installMain`'s `noninteractive()` function will call `process.exit(1)` with an error if neither a `skillset` parameter nor an existing `activeSkillset` in config is present — this situation arises on a fresh install where no skillset has ever been configured. After the switch completes, the command persists `activeSkillset` to config via `updateConfig()` at the command layer, gated on `resolved.source !== "cli"` to avoid persisting state from transient CLI overrides. When the `redownloadOnSwitch` config setting is not `"disabled"`, the command provides an `onRedownload` callback to the flow that calls `registryDownloadMain` to re-download the skillset from the registry before switching. The command also provides an `onReadFileDiff` callback that reads the current installed file from `<installDir>/.claude/<relativePath>` and the original profile source from `~/.nori/profiles/<skillsetName>/`, mapping installed paths back to profile source paths (e.g., `commands/` maps to `slashcommands/`, `agents/` maps to `subagents/`, `skills/` maps directly). For `agents/` paths, the callback checks for a flat file in `subagents/` first, then falls back to a directory-based subagent at `subagents/<name>/SUBAGENT.md`. For `.md` files, the original content has template path substitution applied via `substituteTemplatePaths` to match what was installed. Files without a direct profile source mapping (e.g., `CLAUDE.md`, `settings.json`) return null, causing the flow to show a "Diff not available" message.

The change detection is delegated to `detectLocalChanges()` from @/src/cli/features/agentOperations.ts, which internally uses the manifest module from @/src/cli/features/manifest.ts. When capturing config before switching, the `onCaptureConfig` callback loops over all default agents and calls `captureExistingConfig` on each, ensuring all agents preserve their current state before the switch.

**watch** (@/src/cli/commands/watch/): Monitors Claude Code sessions by tailing transcript files. Supports `watch` (start) and `watch stop` (stop daemon).

**clear** (@/src/cli/commands/clear/): Removes all Nori-managed agent configuration from the install directory and clears the `activeSkillset` from config. The `clearMain` function loads the config, resolves the install directory via `resolveInstallDir()`, iterates all default agents (via `getDefaultAgents()`), and calls `removeSkillset({ agent, installDir })` on each. Then saves config with `activeSkillset: null` while preserving all other config fields (auth, transcript settings, etc.). Returns early with an info message if no config exists. Unlike `factory-reset`, this command only removes Nori-managed files (using the manifest for precision) rather than wiping the entire agent directory, and does not require confirmation since the operation is reversible by re-switching to a skillset.

**clear-current** (@/src/cli/commands/clear-current/clearCurrent.ts): Walks from the current working directory (or optional `dir` argument) upward to the filesystem root and removes Nori-managed agent configuration from every directory where a `.nori-managed` marker is detected. Unlike `clear`, this command does NOT clear `activeSkillset` from config and does NOT use `resolveInstallDir()` — it operates purely on the directory tree relative to CWD. Uses `isInstalledAtDir` for detection and `removeSkillset` from @/src/cli/features/agentOperations.ts for removal; because manifests are keyed per (agent, install dir) and `removeManagedFiles` drops the marker even without a manifest, clearing multiple directories succeeds in any order with no extra cleanup at the command layer. Registered as an unframed command (no `wrapWithFraming`) with a hidden alias `cc`. The parent-walking pattern mirrors `findActiveSkillsets` in @/src/cli/commands/list-active/listActive.ts.

**link** (@/src/cli/commands/link-skillset/linkSkillset.ts): Creates a symlink from `~/.nori/profiles/<name>` to a target directory, enabling development workflows where the skillset source lives in an external git repo. Name resolution: uses `--name` if provided, then reads the target's `nori.json` name, then falls back to the directory basename. Supports org-scoped names (e.g., `org/name`) by creating intermediate parent directories. Validates that the target exists and is a directory, and that no skillset already exists at the link path (checked via `fs.lstat` to detect both real dirs and existing symlinks). Returns `CommandStatus` without framing.

**unlink** (@/src/cli/commands/unlink-skillset/unlinkSkillset.ts): Removes a symlinked skillset from `~/.nori/profiles/`. Safety guard: uses `fs.lstat` and refuses to delete real directories (only removes symlinks). If the unlinked skillset was the active skillset in config, clears `activeSkillset` via `updateConfig({ activeSkillset: null })`. Config errors during this cleanup are swallowed to avoid blocking the unlink operation.

**completion** (@/src/cli/commands/completion/): Generates shell completion scripts for Bash and Zsh. `completionMain` routes the `<shell>` argument to `generateBashCompletion()` or `generateZshCompletion()`, which return static script strings printed to stdout. Users activate completions via `eval "$(nori-skillsets completion bash)"` or the zsh equivalent. The generated scripts handle static completions (subcommands, per-command flags) inline, and dynamic completions for `switch` by calling back to `nori-skillsets list 2>/dev/null` at tab-completion time. All three binary names (`nori-skillsets`, `nori-skillset`, `sks`) are registered for completion. Hidden command aliases (`switch-skillset`, `switch-skillsets`, `list-skillsets`, `list-skillset`, `fork-skillset`, `edit-skillset`) are excluded from completion candidates.

**fork** (@/src/cli/commands/fork-skillset/forkSkillset.ts): The `forkSkillsetMain` function copies an existing skillset directory to a new name under `~/.nori/profiles/`. Validates the source is a valid skillset by checking for `nori.json` (using the same `MANIFEST_FILE` constant from @/src/norijson/skillset.ts used by `listSkillsets()`), validates the destination does not already exist, creates parent directories for namespaced profiles (e.g., `org/name`), and copies recursively via `fs.cp`. After copying, reads the destination's `nori.json` via `readSkillsetMetadata()`, updates the `name` field to the new skillset name (the full path including namespace, e.g., `myorg/forked-profile`), and writes it back via `writeSkillsetMetadata()`. This ensures the forked skillset's metadata reflects its new identity. The primary command is `fork` with a hidden alias `fork-skillset`, following the same alias pattern as other commands.

**new** (@/src/cli/commands/new-skillset/newSkillset.ts): The `newSkillsetMain` function creates a new empty skillset interactively, prompting for metadata using `newSkillsetFlow` from @/cli/prompts/flows. The flow collects name (required), description, license, keywords (comma-separated), version, and repository URL. Validates skillset names (allows namespaced names like `org/name` via `validateSkillsetName` which enforces at most one namespace level). Returns null on cancellation. The command validates the destination does not already exist (via `fs.access`), creates parent directories for namespaced profiles, and writes `nori.json` via `writeSkillsetMetadata()` with all collected metadata including `type: "skillset"`. Optional fields (description, license, keywords, repository) are omitted from nori.json if null. Repository URLs are stored as plain strings in the `NoriJson` manifest. Version defaults to `"1.0.0"` if not provided. The `nori.json` file serves as the skillset marker for `listSkillsets()`. The primary command is `new` with a hidden alias `new-skillset`. Follows npm-init-style workflow for discoverability of metadata fields.

**import-mcp** (@/src/cli/commands/import-mcp/importMcp.ts): Scans known on-disk MCP config locations across the user's machine (Claude `.mcp.json` / `~/.claude.json`, Codex `~/.codex/config.toml` and `<cwd>/.codex/config.toml`, Gemini `<cwd>/.gemini/settings.json` and `~/.gemini/settings.json`, Cursor `<cwd>/.cursor/mcp.json` and `~/.cursor/mcp.json`, VS Code `<cwd>/.vscode/mcp.json`), parses each via `parseAgentConfig` from @/src/cli/features/shared/mcpEmitter.ts, sanitizes literal-looking secrets to `${env:VAR}` placeholders, dedupes by server name (first match wins), writes one canonical JSON file per server into `~/.nori/profiles/<skillset>/mcp/<name>.json`, and merges the auto-derived environment variable list into `nori.json` `requiredEnv` (preserving any user-authored `description`/`url` enrichments across re-imports). The skillset name argument is optional; in interactive mode the command lists available skillsets via `listSkillsets()` and presents a `select()` prompt. Non-interactive mode requires the argument. See @/src/cli/commands/import-mcp/docs.md.

**register** (@/src/cli/commands/register-skillset/registerSkillset.ts): The `registerSkillsetMain` function creates `nori.json` for an existing skillset folder under `~/.nori/profiles/`. When invoked without arguments, reads the current active skillset from config using `loadConfig()` + `getActiveSkillset()`. When given an optional `name` argument, operates on the specified skillset. Validates the directory exists and does not already have `nori.json` (via `fs.access` checks). Collects metadata interactively using `registerSkillsetFlow` from @/cli/prompts/flows, which prompts for description, license, keywords (comma-separated), version, and repository URL. The name field is derived from the folder basename rather than being collected from the user. Writes `nori.json` via `writeSkillsetMetadata()` with all collected metadata including `type: "skillset"`. Optional fields (description, license, keywords, repository) are omitted from nori.json if null. Repository URLs are stored as plain strings in the `NoriJson` manifest. Version defaults to `"1.0.0"` if not provided. The primary command is `register` with a hidden alias `register-skillset`. This command is useful for existing skillset folders created before the `nori.json` manifest was introduced, or for manual directory creation workflows.

**config** (@/src/cli/commands/config/config.ts): Configuration of `defaultAgents` (array of agents), `installDir`, `redownloadOnSwitch`, and `claudeCodeStatusLine` settings, supporting both interactive and non-interactive modes. The config command passes `agentOverride: null` explicitly to `getDefaultAgents()` to suppress the global `--agent` override, because it manages the agent set itself via its own `--agents` (plural) option. Returns a `CommandStatus` (from `@/cli/commands/commandStatus.ts`) indicating success or cancellation. `configMain` accepts optional args: `agents` (comma-separated string), `installDir`, `redownloadOnSwitch` (boolean), `claudeCodeStatusLine` (boolean), and `nonInteractive`. When any config-specific option is provided (or `nonInteractive` is set), the command bypasses `configFlow` entirely and applies values directly via `updateConfig()` -- no interactive prompts run and no post-save install/cleanup prompts are shown. The `parseAgents` helper splits the comma-separated `--agents` value, trims whitespace, filters empties, and validates each name against `AgentRegistry.getInstance().get({ name })`, which throws for unknown agents. When `nonInteractive` is set but no options are provided, the command throws an error listing the available flags. In interactive mode (no args), runs `configFlow` to collect user input via multiselect for agents, install directory, a redownload-on-switch toggle, and a Claude Code status line toggle, then saves the updated fields via `updateConfig()`, which preserves all other config fields automatically. Agent options are resolved dynamically from `AgentRegistry` and carry each agent's `supportTier`, which the flow uses to list supported agents first and mark experimental ones in the hint. After saving in interactive mode, the command detects changes to `installDir` and `defaultAgents` by comparing old vs new values. All prompts are skipped when there is no active skillset. When `installDir` changes: prompts to install the active skillset at the new directory, then prompts to clean up the old directory. Cleanup uses `AgentRegistry.getInstance().getAll()` to find all registered agents, filters by `isInstalledAtDir({ agent, path: oldInstallDir })` to identify which agents are actually installed at the old directory, and calls `removeSkillset({ agent, installDir })` on each. The ordering matters: cleanup runs before install so the manifest reflects the old directory during removal. When `defaultAgents` changes (and `installDir` did not change): separately handles removed and added agents. For removed agents, prompts to clean up their managed configuration at the current `installDir` via `removeSkillset()`. For added agents, prompts to install the active skillset for the new agents. The `defaultAgents` branch only fires when `installDir` did not also change (the installDir flow already triggers a full reinstall). At the registration layer in `noriSkillsetsCommands.ts`, the config command defines `--agents <agents>`, `--redownload-on-switch` / `--no-redownload-on-switch`, and `--claude-code-status-line` / `--no-claude-code-status-line` options, and forwards global opts (`installDir`, `nonInteractive`, `silent`) to `configMain`. The `--install-dir` option is reused from the global program-level definition rather than duplicated on the command.

**factory-reset** (@/src/cli/commands/factory-reset/factoryReset.ts): The `factoryResetMain` function removes all configuration for a given agent. It blocks non-interactive mode (prints an error and returns), looks up the agent by name via `AgentRegistry.getInstance().get({ name })`, checks that the agent declares `getArtifactPatterns` (agents without it do not support factory reset), and runs `factoryResetFlow` with callbacks that discover artifacts via `findArtifacts` from @/src/cli/features/agentOperations.ts and delete them. Defaults `path` to `process.cwd()` if not provided. This follows the same pattern as `switch` for refusing destructive operations in non-interactive mode.

**dir** (@/src/cli/commands/dir/dir.ts): Opens the Nori skillsets directory (`~/.nori/profiles`) in the system file explorer. In non-interactive mode, outputs the plain path to stdout for scripting use. In interactive mode, spawns the platform-specific open command (`open` on macOS, `xdg-open` on Linux) as a detached process so the CLI can exit immediately while the file explorer remains open. Falls back to printing the path if spawning fails. Uses `@clack/prompts` output (`log.success`, `log.step`) for interactive feedback.

**current** (@/src/cli/commands/current-skillset/currentSkillset.ts): Displays the currently active skillset name. The command has two forms: `current` (primary, shorthand) and `current-skillset` (hidden alias for backward compatibility). Uses `loadConfig()` to read the centralized config, then `getActiveSkillset({ config })` to retrieve the active skillset. Outputs the skillset name via `raw()` logger for easy parsing in scripts. Exits with error if no config exists or no active skillset is configured.

**edit-skillset** (@/src/cli/commands/edit-skillset/editSkillset.ts): The `editSkillsetMain` function opens a skillset's skillset directory in VS Code, or falls back to printing the path and directory contents with manual instructions. When given an optional `name` argument, it opens that specific profile; otherwise it reads the active skillset from config via `loadConfig()` + `getActiveSkillset()`. The skillset directory is resolved via `getNoriProfilesDir()` (i.e., `~/.nori/profiles/<skillsetName>/`). VS Code launch uses `child_process.execFile('code', [path])`. Supports the `--agent` flag for multi-agent setups and namespaced skillset names (e.g., `myorg/my-profile`).

### Things to Know

Command implementations follow a consistent pattern: the subdirectory exports a `*Main` function containing the business logic, while `noriSkillsetsCommands.ts` is the canonical Commander registration surface. All `*Main` functions return `CommandStatus` (from `@/cli/commands/commandStatus.ts`) with `success`, `cancelled`, and `message` fields. Commands do not call `intro()` or `outro()` from `@clack/prompts` directly; visual framing is the responsibility of the top-level caller. The `noriSkillsetsCommands.ts` file uses its own registration wrappers rather than calling per-command registration helpers directly, because it adds global option forwarding (e.g., `installDir`, `nonInteractive`, `silent`). The `install`, `download`, and `download-skill` commands all forward `nonInteractive` and `silent` from `globalOpts` (extracted via `program.opts()`) to their respective `*Main` functions, which in turn propagate them to interactive flows. The coercion pattern is `nonInteractive ?? silent ?? false` at the boundary where a boolean is needed, so `--silent` implies non-interactive behavior.

**Agent resolution via global override:** The `--agent` flag is resolved centrally by a `preAction` hook in `@/src/cli/nori-skillsets.ts` that calls `setGlobalAgentOverride()` before every command action. Most commands simply call `getDefaultAgents({ config })` without passing an explicit `agentOverride`, and the global override is applied automatically. Commands that define their own local `--agent` option (e.g., `switch`, `edit`, `current`, `watch`) pass `options.agent` as the explicit `agentOverride` parameter, which takes precedence over the global override. The `config` command passes `agentOverride: null` to suppress the global override because it manages the agent set via its own `--agents` (plural) option. Tests that simulate the full CLI must add the `preAction` hook to their test Commander program (see `switchSkillset.test.ts` for the pattern).

- **Config persistence is the command layer's responsibility:** The `installDir` field in `.nori-config.json` has a single writer: `sks config`. The `activeSkillset` field is persisted by the command layer (`switch-skillset`, `registry-install`) via `updateConfig()`, gated on the install directory provenance: when `resolved.source === "cli"`, config persistence is skipped to prevent transient `--install-dir` overrides from writing state to `.nori-config.json`. The features-layer `switchSkillset()` operation only validates and logs -- it does not call `updateConfig()`. All commands use `resolveInstallDir()` at runtime to determine the operational install directory (CLI `--install-dir` flag > config > home dir) but never write `installDir` back to config. When the install orchestration's `noninteractive()` function (in @/src/cli/features/install/install.ts) needs to pass a runtime `installDir` override to `completeInstallation()`, it overlays it on the reloaded config object (`{ ...config, installDir: normalizedInstallDir }`) rather than persisting it.
- **Manifests are keyed per (agent, install dir):** Manifest operations run unconditionally, including for `--install-dir` overrides. Because @/src/cli/features/manifest.ts keys manifests by both agent and install directory, an override directory gets its own manifest and real local-change detection instead of false positives against another directory's state. The former `skipManifest` flag (threaded through install/switch/registry-install to protect override dirs from the old global per-agent manifest) is gone.
- **Storage buckets:** Local skillsets live in on-disk buckets under `~/.nori/profiles/` (`personal/` for locally created, `public/` for public-registrar downloads; org skillsets stay at `<orgId>/<name>`), while their user-facing names remain bare. Commands never hardcode `profiles/<name>` anymore: create-path commands (`new`, `fork`, `external --new`) place new skillsets via `skillsetCreateDir`, and read-path commands (`switch`, `edit`, `register`, `upload`, `install`, `skill-download`, `subagent-download`, `skill-upload`, `import-mcp`) resolve a bare name via `resolveSkillsetDir` (both from @/src/norijson/skillset.ts). A one-time startup migration relocates legacy flat profiles into buckets — see @/src/norijson/docs.md and @/src/cli/docs.md.
- There is no `install` subdirectory here: the install orchestration (including its ASCII banners) lives in the features layer at @/src/cli/features/install/ and is imported statically by `switch-skillset`, `config`, and `registry-install`.
- The upload/download commands do not hand-roll tar pipelines, atomic swaps, or `.nori-version` handling — those primitives have a single owner in @/src/packaging/ (gzip sniffing, extraction, atomic replace with restore-on-failure, provenance, and shared registry lookup/formatting).
- The `skill-download` command (@/src/cli/commands/skill-download/) downloads individual skills and updates both `skills.json` and `nori.json` manifests in the target skillset.
- The `skill-upload` command (@/src/cli/commands/skill-upload/) publishes a single skill from `~/.nori/profiles/<skillset>/skills/<skill>` to the registry, complementing the whole-skillset `registry-upload` path. Registered as `upload-skill` and wired via `registerNoriSkillsetsUploadSkillCommand` using `wrapWithFraming`. It targets the pre-existing `registrarApi.uploadSkill` endpoint (`PUT /api/skills/:skillName/skill`) whose server response does not emit `SkillCollisionError`, so collision handling is done client-side: the command fetches the latest remote tarball, extracts `SKILL.md`, and byte-compares against the local copy to decide between short-circuit "already up to date", silent publish, or an interactive bump/diff/cancel prompt. Skills with `nori.json.type: "inlined-skill"` are blocked from upload since they belong to a parent skillset. The `--description` flag defaults to the skill's local `nori.json.description`, and a successful upload auto-syncs the bumped version back to the local `nori.json`. See @/src/cli/commands/skill-upload/docs.md.
- The `subagent-download` command (@/src/cli/commands/subagent-download/) downloads individual subagent packages. Unlike skills, subagents are "flattened" during installation: the full directory structure is persisted in the skillset profile at `subagents/<name>/`, but only `SUBAGENT.md` content (with template substitution) is written to each agent's `agents/<name>.md`. Subagent dependencies are tracked only in `nori.json` (`dependencies.subagents`), with no `skills.json` equivalent.
- The `external` command (@/src/cli/commands/external/) installs skills directly from GitHub repositories. It clones the repo, discovers SKILL.md files, and installs them following the same dual-installation pattern as `skill-download` (live copy to `~/.claude/skills/` with template substitution, raw copy to skillset's `skills/` directory). Supports `--new <name>` to create a brand-new skillset and install skills into it in a single step, or `--skillset <name>` to target an existing one. Writes a `nori.json` provenance file (instead of `.nori-version`) to track the GitHub source URL, ref, subpath, and installation timestamp. See @/src/cli/commands/external/docs.md for details.
- The `registry-install` command combines `registry-download` with install/switch to provide a single-step "download and activate" flow from the registrar (public or org). It broadcasts installation and switching to all configured `defaultAgents`. It derives the **user-facing** skillset name by parsing the spec with `parseNamespacedPackage` and mapping it via `namespacedName` (both from @/src/utils/url.ts): a bare `<name>` or `public/<name>` reference yields the bare name `<name>`, while `<org>/<name>` keeps its namespace. That name is what gets recorded as `activeSkillset` and handed to install/switch, so `sks current` stays consistent with `sks list`. The on-disk directory differs: `registryDownloadMain` stores public packages in the `public/` bucket (via `namespacedOnDiskName`), and downstream resolution of the bare name happens through `resolveSkillsetDir` from @/src/norijson/skillset.ts (its `checkLocalSkillsetExists` fallback also uses `resolveSkillsetDir`). A malformed spec short-circuits with an "Invalid skillset specification" failure. It snapshots `isFirstTimeInstall` via `hasExistingInstallation()` **before** calling `registryDownloadMain()`, because the download step may auto-init and create `.nori-config.json`, which would cause `hasExistingInstallation()` to return true afterward -- making the first-time install path (welcome banners via `installMain`) unreachable. Config persistence (`updateConfig({ activeSkillset })`) is gated on `resolved.source !== "cli"` to prevent transient CLI overrides from writing to `.nori-config.json`.
- The `registry-search` command (@/src/cli/commands/registry-search/) searches for skillsets, skills, and subagents in parallel across registries. It always queries the public registry without authentication; if org auth is configured, it also searches the org registry with authentication (org results displayed first). Results are grouped into three sections ("Skillsets:", "Skills:", "Subagents:") in the output. Non-API errors (e.g., auth failures) are silently hidden per search type; network errors are surfaced. Download hints at the bottom dynamically include only the entity types that had results (e.g., `download-subagent` hint only appears when subagent matches exist). The `SearchFlowResult` carries separate `skillsetCount`, `skillCount`, and `subagentCount` fields for the summary status message.
- The skillset upload implementation (`registry-upload/registryUpload.ts`) creates a gzipped tarball from the local skillset directory via `createArchive` from @/src/packaging/archive.ts, which filters entries through the shared `shouldExcludeFromUpload` helper at `@/src/utils/uploadFileFilter.ts` so editor swap files, OS junk, `.nori-version`, and dependency/build "bloat" directories (`node_modules/`, `.venv/`, and Cargo-adjacent Rust `target/` output, detected via `collectCargoManifestDirs`) never reach the registry's content hash. It is exposed through the public `upload` command registered in `noriSkillsetsCommands.ts`, including CI-oriented options such as `--non-interactive` and `--resolve <strategy>`. The authorization flow evaluates branches in order: explicit registry URL, public org (any authenticated user -- no org membership required), org-scoped (requires org membership), and unauthenticated (error). The "public" org is treated as an open namespace where any authenticated user can upload; server-side moderation (`vouchStatus: 'pending'`) gates non-admin uploads rather than client-side org membership. This branch order mirrors the download commands. Skill collision errors from the API include conflict metadata that enables auto-resolution for unchanged skills. After a successful upload, `syncLocalStateAfterUpload` writes the uploaded version and `registryURL` back to the local `nori.json`, writes a `.nori-version` file, and updates extracted/linked skill versions in `metadata.dependencies.skills`. This sync is wrapped in try/catch so failures produce a warning but never mask a successful upload. Dry-run mode skips the sync entirely. See @/src/cli/commands/registry-upload/docs.md for the full authorization branch order.
- Google OAuth uses Desktop app client credentials; the client secret is not truly secret (same as firebase-tools, gcloud CLI, etc.). CSRF protection uses a cryptographic nonce (`generateState()`) verified in the callback.
- **login** (@/src/cli/commands/login/): Authenticates against `noriskillsets.dev` via Firebase (email/password or Google SSO) OR persists an API token for a private-org registrar. All three modes are reachable from the interactive `select` menu (Email / Password, Google SSO, API Token) or from flags (`--token`, `--google`, `--email`/`--password`). The `--token <value>` flag short-circuits all Firebase flows and writes `auth.apiToken` and `auth.organizationUrl = https://{orgId}.noriskillsets.dev` to config, clearing `username`/`refreshToken`/`password`. The flag path and the interactive "API Token" path share a single module-private `loginWithApiToken` helper, so validation, config mutation, and the Firebase-session-overwrite warning behave identically across entry points. The orgId is parsed from the token (format `nori_<orgId>_<64hex>`) via `extractOrgIdFromApiToken` from @/src/utils/apiToken.ts; there is no separate `--org` flag. Rejects tokens whose embedded org is `public` (server never accepts API tokens on the public apex), validates the token via `isValidApiToken`, and is mutually exclusive with `--email`/`--password`/`--google` on the flag path. See @/src/cli/commands/login/docs.md.
- `publicUploadGuard.ts` is a shared upload helper exporting `guardPublicUpload`, called by both `registry-upload` and `skill-upload` before auth resolution. It refuses to publish to the public registry (`noriskillsets.dev`) unless the caller made the target explicit — via the `--public` flag, a `public/<name>` namespace, or `--registry <url>` — failing hard in silent/non-interactive mode and prompting (default No) interactively. A bare, non-namespaced name defaults to public and trips the guard. This is a root-cause fix for an incident where an org-intended package was silently published publicly.
- The registry-facing download/search/upload commands (`registry-download`, `registry-search`, `registry-upload`, `skill-download`, `skill-upload`, `subagent-download`) use the pure credential helpers from @/src/api/authCredentials.ts (`hasRegistryAuthCredentials`, `toRegistryAuth`) to decide whether `config.auth` has usable registry credentials and to construct `RegistryAuth` objects. Usable config credentials are `refreshToken`, `apiToken`, or an unexpired broker-managed `idToken`; matching `NORI_API_TOKEN` values can also authorize env-only flows. For org-scoped namespaces, the shared membership check + registry URL derivation + `RegistryAuth` construction is `resolveOrgRegistryAuth` from @/src/core/registryAuthResolution.ts; each command keeps its own surrounding policy (env-token matching, anonymous public downloads, error strings). The commands then thread the resulting auth through `getRegistryAuthToken`, which owns per-registry token precedence and org-scoping.
- Installation manifests are written per (agent, install dir) at `~/.nori/manifests/<agentName>/<dirKey>.json` (see @/src/cli/features/manifest.ts). `installSkillset` in @/src/cli/features/agentOperations.ts scopes tracking via `getManagedFiles({ agent })` and `getManagedDirs({ agent })`.
- Manifest writing failures are non-fatal and do not block installation.
