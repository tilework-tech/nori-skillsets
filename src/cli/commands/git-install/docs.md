# Noridoc: git-install

Path: @/src/cli/commands/git-install

### Overview

- Implements the consumer-side Git source for `sks install <slug> --from <remote>`, including optional historical selection with `--pin <full-sha>`.
- Derives the required source branch `refs/heads/skillsets/<slug>` from the requested local name, so users provide only the remote and slug. A same-name tag is not accepted in place of that branch.
- Acquires and validates the selected branch tip or historical commit in one command-level flow before handing the canonical local identity `personal/<slug>` to the shared activation pipeline.

### How it fits into the larger codebase

- The central command registration layer dispatches here only when `--from` is present. An install without `--from` continues through the existing Registrar path, and failures or user cancellation in this path are returned directly rather than triggering a Registrar fallback.
- Git acquisition is deliberately independent of Registrar APIs. It does not download dependencies or apply a dependency-materialization heuristic.
- Successful acquisition produces a normal local skillset directory, so activation reuses the same agent registry, install-directory resolution, and shared non-interactive installer as other local skillsets.
- The source checkout remains a Git working tree under the `personal/` storage bucket. This feature does not add custom provenance, durable trust state, update behavior, or activation rollback.

### Core Implementation

- The requested slug deterministically selects `skillsets/<slug>` and the destination `personal/<slug>`. The root `nori.json` must have exactly that slug as its `name` and `skillset` as its `type`; the remote cannot choose or override the local identity.
- Source approval happens before acquisition. Interactive installs require affirmative confirmation, while non-interactive and silent installs require `--trust-source`. Approval is now persisted as durable trust (Feature 4): an approved `(canonicalized remote, skillsets/<slug>)` is recorded in `~/.nori/trust.json` and skips the gate on later installs; see @/src/cli/commands/trust/docs.md. URL-form userinfo is redacted regardless of URL scheme. Query-key matching percent-decodes and compatibility-normalizes the key before recognizing token, OAuth, client-secret, signing, and password parameters; both ampersand and semicolon separators are supported. The same sanitizer protects trust prompts and surfaced Git errors.
- Git remote-helper syntax (`<transport>::<address>`) and URL schemes outside the explicit `http`, `https`, `ssh`, `git`, `git+ssh`, and `file` allowlist are rejected before approval because they can invoke arbitrary transport helpers. Local paths and SCP-style SSH remotes remain supported. Declining or cancelling a supported source returns a cancelled command status rather than an error.
- Git-backed installation requires Git 2.29 or newer. Acquisition uses native Git with argument-array process execution, a 60-second timeout, and a shared status/error path. It removes inherited Git repository-routing variables (`GIT_DIR`, `GIT_WORK_TREE`, `GIT_COMMON_DIR`, `GIT_INDEX_FILE`, `GIT_OBJECT_DIRECTORY`, `GIT_ALTERNATE_OBJECT_DIRECTORIES`, and `GIT_SHALLOW_FILE`) while preserving ordinary non-prompting authentication behavior. Non-interactive and silent execution sets `GIT_TERMINAL_PROMPT=0` and enforces OpenSSH `BatchMode=yes`, so Git credential, SSH host-key confirmation, and SSH password/passphrase challenges fail instead of opening a prompt. Both an inherited `GIT_SSH_COMMAND` and a configured OpenSSH-compatible `core.sshCommand` keep unrelated options, receive batch mode when needed, and are rejected if they explicitly disable it. `GIT_SSH` executables, wrappers, and clients that cannot accept OpenSSH options are outside the compatibility contract.
- Acquisition atomically reserves the final destination, normalizes a relative local remote once to an absolute path, uses `ls-remote --heads` to preflight the exact `refs/heads/skillsets/<slug>` ref, initializes a repository at the destination, and fetches only that branch into its remote-tracking ref with tags disabled, `--no-write-fetch-head`, no client-imposed depth limit, and `--update-shallow`. This prevents a same-name tag or another ref from standing in for the required branch, avoids retaining a credential-bearing acquisition URL in `FETCH_HEAD`, retains complete branch history from a normal source, and permits the fetch to accept the advertised boundary of a source repository that is itself shallow. The fetched tracking ref, rather than the earlier `ls-remote` observation, supplies the authoritative branch tip for checkout and pin ancestry, so a branch movement between the two network operations is handled consistently. The same normalized remote is stored as `origin`; that ordinary metadata removes passwords, every query parameter and fragment, and HTTP(S)/file usernames.
- Both modes fetch the exact branch ref without a client-imposed depth limit. Without `--pin`, acquisition checks out the fetched tracking-ref tip with the branch attached and does not claim complete ancestry. With `--pin`, it accepts only a 40-character SHA-1 or 64-character SHA-256 hexadecimal object ID, requires the supplied ID to equal Git's fully resolved object ID, rejects the source if Git reports a shallow repository, verifies that the commit is reachable from the fetched branch tip through complete all-parent history, and checks it out with detached `HEAD`. Comparing the supplied and resolved IDs prevents a 40-character SHA-256 abbreviation from being treated as a full pin.
- Object inspection distinguishes a missing object from a failed Git operation. Missing pins receive the branch-history error, while corruption, permission, process, or malformed-output failures remain Git errors rather than being rewritten as user input errors.
- Validation parses NUL-delimited `git ls-files --stage -z` records before reading metadata, so unusual tracked pathnames cannot alter the safety checks. For every tracked path it normalizes the first root component with compatibility normalization, case folding, and removal of the code points Git ignores on HFS filesystems. It rejects symbolic links, submodules, and any normalized root component equal to `.nori-version`, including reserved directory aliases with content below them. It likewise rejects manifest aliases and manifest directories, requiring exactly one compatibility-equivalent manifest entry: the exact lowercase root `nori.json` tracked as a regular file. This prevents filesystem aliases, colliding entries, or manifest symlinks from becoming the authority. Acquisition and validation errors remove the reserved destination, leaving no partial local skillset.
- Validation runs after commit selection, so a pinned install validates the selected historical tree rather than the branch tip. Every configured default agent is resolved before destination reservation, so an invalid selection cannot leave a checkout behind.
- After validation, each agent is activated through the existing shared non-interactive installer with persistence of the target identity and install marker deferred. The Git command writes that agent's `.nori-managed` marker only after its activation succeeds, then commits `personal/<slug>` to global config exactly once and only after every configured agent succeeds. An explicit install-directory override makes the operation transient and skips that final config commit while retaining the successful per-agent markers. This command-level ordering does not add rollback or a shared transactional-activation mechanism.
- The Git command locally buffers console, stdout, and stderr output from each nested activation. It discards that output when activation or the final target-identity commit fails, then replays it in order after overall success for both global and explicit-install-directory installs. Silent mode discards the buffer even on success. This behavior is confined to the Git command; it does not change shared installer semantics, snapshot managed files, or satisfy shared transactional activation.

### Things to Know

- Local-name collisions fail closed: an existing `personal/<slug>` path is never overwritten or reused, including when it is a symlink or an incomplete checkout.
- Exact-ref acquisition imposes no client-side depth limit. A normal source therefore retains its complete branch history; `--update-shallow` permits an unpinned install to accept the advertised boundary of a source repository that is itself shallow without claiming complete ancestry. Historical pinning requires Git to report a non-shallow repository before proving ancestry. Because this path initializes and fetches an exact ref instead of invoking `git clone`, user-level `clone.rejectShallow` configuration does not alter either outcome.
- The checkout is a one-time snapshot at installation time. There is no automatic fetch, fast-forward, or session-restart update behavior; explicit fast-forward updates are handled by `sks update` (see @/src/cli/commands/update-skillset/docs.md). Detached `HEAD` remains the only pin marker — the source is read from native Git metadata rather than a sidecar.
- Successful pinned installs report the resolved full object ID. Abbreviated IDs (including 40-character abbreviations in SHA-256 repositories), refs, tags, revision expressions, missing objects, non-commit objects, commits outside the selected branch's all-parent history, and pinned sources whose complete history cannot be proven are rejected.
- Validation intentionally stops at manifest identity/type and tracked-entry safety. Dependency declarations are not resolved through the Registrar and are not checked for matching materialized content by this command.
- Acquisition and validation failures clean up the checkout. Activation failures retain the validated checkout and return a recovery command that preserves an explicit install-directory override and the effective single-agent scope. Dynamic recovery arguments are POSIX-shell-quoted so copy/paste replays paths and identities literally without shell expansion. If a later agent fails, managed files and the new marker written for an earlier successful agent may remain, while the failing agent's previous or absent marker is preserved; the global target-identity commit is not attempted. No managed files are rolled back.
- A failure of the final target-identity commit is distinct from an activation failure: every configured agent has already activated and received its new marker, the previous global `activeSkillset` remains selected, and the validated checkout is retained. The reported recovery command can be rerun after correcting the config-write problem; it does not imply that completed managed files were rolled back.
- Feature 2 directly owns exact historical commit acquisition. Its co-delivered changes to the shared tip/pin path are limited to authority, validation, diagnostics, unattended execution, and successful silent activation; they do not add update metadata, durable trust, transactional activation, or silent failure semantics.
- OpenSSH `BatchMode=yes` is the v1 SSH contract. `GIT_SSH` executables, arbitrary `GIT_SSH_COMMAND` wrappers, Plink/TortoisePlink, and askpass compatibility are outside this feature.
- The Git process boundary closes child stdin for every command. Treating an `EPIPE` caused by Git exiting before consuming that input as a normal early close is an internal reliability measure, not a separate user-facing capability; other stdin failures remain part of the shared Git error result.
- The checkout contains sanitized ordinary Git origin and branch metadata but no Nori-specific source provenance or durable trust marker. SSH-style usernames may remain because they select a remote account rather than authenticate it. The sanitized origin is command-local credential hygiene, not durable update-source or authentication metadata; Features 3 and 4 own those policies. The checkout never carries Registrar `.nori-version` provenance and is never eligible for Registrar fallback behavior.
- The update adapter and the native-Git source read-model live in @/src/cli/features/gitPackage.ts and are consumed by `sks update`; installation does not persist any durable source sidecar.
