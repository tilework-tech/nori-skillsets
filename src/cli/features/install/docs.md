# Noridoc: install

Path: @/src/cli/features/install

### Overview

The install orchestration: the end-to-end pipeline that initializes the Nori environment, resolves the active skillset, runs each agent's feature loaders, and displays completion banners. This is not a registered CLI command -- it is the shared engine that commands invoke when they need a full (re)install. It previously lived under `src/cli/commands/install/` and was moved into the features layer so commands can import it statically without module cycles.

### How it fits into the larger codebase

- Called statically by `switch-skillset` (silent reinstall after a switch), `config` (reinstall on installDir/agent changes), and `registry-install` (download-and-activate). Git-backed installation calls the non-interactive core after establishing its own command-level silent state. The old dynamic `import()` calls that existed solely to dodge the switch -> install -> init module cycle are gone; the dependency direction is now commands -> features -> utils throughout.
- `initialize.ts` exports `ensureNoriInitialized`, the non-interactive init core. The `init` command (@/src/cli/commands/init/init.ts) delegates its non-interactive path to it; the install orchestration calls it directly instead of invoking the init command, which is what broke the cycle.
- Delegates skillset installation to `installSkillset` from @/src/cli/features/agentOperations.ts and agent lookup to @/src/cli/features/agentRegistry.ts.
- Fires install-lifecycle analytics via @/src/cli/installTracking.ts and reads/writes config via @/src/cli/config.ts.
- `scripts/build.sh` chmods the built `install.js` at this location.

### Core Implementation

- `install.ts` exposes `main` (establishes optional silent mode around `noninteractive`) and `noninteractive` (init via `ensureNoriInitialized`, resolve skillset, then `completeInstallation`). Both accept an optional `persistActiveSkillset` flag (default: persist). Persisting the global `activeSkillset` to `.nori-config.json` via `updateConfig` is now gated on `persistActiveSkillset !== false`: a transient `--install-dir` switch passes `false` so it never clobbers the user's global active skillset, but the selected skillset is still threaded in-memory into `completeInstallation`'s config (`{ ...config, installDir, activeSkillset: selectedSkillset, persistActiveSkillset: false }`) so the correct files still land in the target dir — and so `configLoader` (which otherwise writes `config.activeSkillset` to disk) keeps the on-disk global value. The non-interactive core also accepts `persistInstallMarkers` (default: persist) so a command can defer `.nori-managed` writes until it knows a particular agent activation succeeded. `completeInstallation` writes a progress marker at `~/.nori-install-in-progress`, runs `installSkillset` for the agent, sends started/completed analytics events, and shows banners.
- `initialize.ts` (`ensureNoriInitialized`) creates `~/.nori/profiles/`, captures pre-existing agent configuration as a `"my-profile"` skillset when no config exists yet, persists `activeSkillset`, and normally writes `.nori-managed` markers (via `markInstall`) for every default agent. The marker's skillset name resolves via `capturedSkillsetName ?? skillset ?? existingConfig?.activeSkillset ?? null`, so callers like `nori switch` can thread the target skillset through. A caller that passes `persistInstallMarkers: false` suppresses these eager marker writes and owns marking each successfully installed agent itself; Git-backed multi-agent activation uses this to avoid relabeling an agent whose activation failed.
- `installState.ts` (`hasExistingInstallation`) detects whether a Nori config file already exists; `registry-install` snapshots this before downloading to decide whether to show first-install banners.
- `installLock.ts` (`withInstallLock`) serializes mutating skillset workflows against one another so two concurrent Nori writers cannot interleave activation, config, or destructive-cleanup writes. It is a reentrant, directory-based filesystem lock at `~/.nori-install.lock`: reentrancy is tracked per async context via `AsyncLocalStorage`, so a command already holding the lock (e.g. `registry-install` calling the shared installer or `registry-download`) does not deadlock on nested acquisition. A stale lock is recovered when its recorded owner PID is dead (Linux additionally verifies the PID's boot-id/start-time identity to survive PID reuse), and an interrupted empty lock directory is reclaimed. Command entry points wrap their body in `withInstallLock`: `install`/`registry-install`, `registry-download`, `switch`, `init`, `config`, `external` install, `clear`, `clear-current`, `factory-reset`, `unlink`, `skill-download`, `subagent-download`, Git install, and direct `updateConfig` calls. Profile-authoring-only commands (`new`, `fork`, `link`) stay outside the lock because they neither activate a skillset nor mutate global config. Read-only version listing (`--list-versions` on the download commands) bypasses the lock so it never creates the Nori home or serializes behind an install. The lock provides mutual exclusion only; it is not a snapshot/rollback transaction and does not restore managed files written before a later failure — that is `activationTransaction.ts`.
- `activationTransaction.ts` (`withActivationTransaction`) makes multi-agent activation atomic: a failure partway through restores the previous usable state instead of leaving a half-activated installation. It nests inside `withInstallLock` (acquiring no lock of its own) and takes a dpkg-style selective backup of exactly the paths activation may overwrite — for each configured agent, its declared managed files/dirs under the agent dir, its instructions file, its `.nori-managed` marker, and its external settings files (`~/.claude/settings.json`) — plus the global `~/.nori-config.json` pointer. It runs the activation `operation`, and on any throw restores every backed-up path (deleting paths that were absent before) before re-raising the original error; on success it discards the backup. The snapshot index is persisted under `~/.nori/.txn/<id>/` with a `committed` marker written on success, so a process killed mid-activation is rolled back by `recoverPendingActivations()`, which runs at the start of the next activation (restore is idempotent). The transaction owns the single `activeSkillset` commit: `registry-install` and `switch` (non-interactive) pass `persistActiveSkillset: false` to the shared installer and commit the pointer once inside the operation, so a failed activation restores the previous pointer too. Not yet wrapped: `git-install` (keeps its forward-recovery model — it retains the checkout and directs the user to re-run `switch --force`, and its checkout acquisition/cleanup is deliberately separate from activation rollback) and the interactive `switch` flow. The `~/.claude.json` mcp `userFile` is not yet snapshotted (it is merge-only and never restored today). It does not model durability (no `fsync`); it protects against activation-failure and process-kill, not power loss.
- Silent mode is a shared install-pipeline contract, not just a framing option. The orchestrator, initialization, agent operations, settings backup, config loader, and visible loader emitters consult `isSilentMode()` before calling Clack, so successful silent activation performs the same filesystem work without status, warning, summary, or loader output. `asciiArt.ts` applies the same guard to the welcome banner and seaweed bed; those use raw `process.stdout.write()` because Clack's `log.*` methods would break their alignment.

### Things to Know

- `noninteractive` calls `process.exit(1)` when no skillset can be resolved (no `skillset` argument and no `activeSkillset` in config), so callers on fresh installs must pass `skillset` explicitly.
- The runtime `installDir` is overlaid on the loaded config (`{ ...config, installDir }`) for operational use but never persisted -- only `sks config` writes `installDir` to `.nori-config.json`.
- Manifest writing always happens inside `installSkillset`; there is no skip flag. Manifests are keyed per (agent, install dir) in @/src/cli/features/manifest.ts, so transient `--install-dir` overrides get their own manifests instead of being skipped.
- Deferring install markers does not make installation transactional. A caller may mark each completed agent accurately, but managed files written before a later failure remain unless a separate source-neutral transaction restores them.
- The agent defaults to `AgentRegistry.getInstance().getDefaultAgentName()` (i.e., `DEFAULT_AGENT_NAME` from @/src/cli/features/agentTable.ts) when no agent is specified.
- Callers that invoke `noninteractive` directly, rather than `main`, own establishing and restoring silent mode for the duration of activation. This keeps the non-interactive core reusable while ensuring every shared emitter observes one consistent process-level state.
